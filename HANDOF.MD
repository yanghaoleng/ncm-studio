# HANDOF

## 当前目标

优化约 200 首音乐的 ZIP 打包体验：降低等待时间、显示按钮百分比，并在卡住或失败时给出重试/分批建议。

## 进度

- [x] 拉取并确认 `main` 与 `origin/main` 一致。
- [x] 转换阶段预计算 CRC32；打包阶段直接写入 ZIP `STORE` 结构并组合原始音频 Blob。
- [x] 新增独立打包模块、重名文件保护、进度与停滞检测。
- [x] 桌面端和移动端按钮显示 0%-100% 进度。
- [x] 20 秒无进度时提示卡顿；捕获异常后提示重试或分批。
- [x] ESLint 和 Vite 生产构建通过。
- [x] Node 模拟 200 首/50 MB：200 个唯一条目，中文名和重名正确，字节一致，进度 100%，耗时约 7 ms。
- [x] Chrome 对比 200 首/100 MB：旧方案约 839 ms，新方案约 59 ms，基准提升约 93%，无控制台错误。
- [x] JSZip 反向解包和系统 `unzip -t` 均验证 200 个条目完整。
- [x] 生产环境已部署到 `https://ncm.mikeywa.icu`，HTTP 200，Chrome 加载无控制台错误。

## 已知边界

- 当前生成经典 ZIP，不含 ZIP64；单包超过 4 GB 时会提示分批。
- 本地没有真实 `.ncm` 样本，本次性能验证使用等价的二进制音频 Blob。
